
## Qiskit Version Upgrade: Impact on HHL Circuit Processing and Solution Quality

### Context

The FVM Euler 1D nozzle solver uses the HHL quantum linear solver at each Newton
iteration, which involves generating a quantum circuit from the linear system,
transpiling it for the target backend, and executing it with shot-based sampling.
These three phases — generation, transpilation, and execution — dominate the
per-case wall time.

Two experiments were run on ORNL Frontier with identical solver parameters but
different Qiskit library versions to measure the effect of upgrading the quantum
SDK on both performance and solution quality.

### Experiments

Both experiments used the same solver configuration:

| Parameter | Value |
|---|---|
| Solver | FVM Euler 1D nozzle |
| Elements | 5 (15 DOF) |
| Time scheme | BDF1 (implicit) |
| CFL | 1e10 (steady-state) |
| Linear solver | HHL (statevector, 150k shots) |
| Newton iterations | 15 |
| Inner iterations | 1 |
| Local time stepping | enabled |
| Trials | 10 |
| Platform | Frontier (ORNL), 1 exclusive node per trial |

The only difference was the library stack:

| Library | Nov 2025 | Apr 2026 |
|---|---|---|
| qiskit | 1.2.4 | 2.3.1 |
| qiskit-aer | 0.15.1 | 0.17.2 |
| qiskit-ibm-runtime | 0.35.0 | 0.46.1 |
| qiskit-algorithms | 0.3.1 | (removed) |
| Python | 3.11 | 3.12.9 |

The Apr 2026 stack uses the forked `quantum_linear_solvers` library ported to
Qiskit 2.x, which drops the deprecated `qiskit-algorithms` dependency.

### Results: Per-Case Wall Time

Mean total quantum processing time per case (sum of circuit generation,
transpilation, and execution across all 15 iterations):

| Phase | Nov 2025 | Apr 2026 | Speedup |
|---|---|---|---|
| Circuit generation | 3795 s (63.3 min) | 909 s (15.2 min) | 4.2x |
| Transpilation | 648 s (10.8 min) | 104 s (1.7 min) | 6.2x |
| Execution (Aer sim) | 926 s (15.4 min) | 576 s (9.6 min) | 1.6x |
| **Total** | **5370 s (89.5 min)** | **1589 s (26.5 min)** | **3.4x** |

### Results: Per-Iteration Timing

Representative single-trial comparison (iteration 0 is the first Newton step,
which constructs the largest circuit due to matrix sparsity pattern):

| Metric | Nov 2025 (per iter) | Apr 2026 (per iter) |
|---|---|---|
| Circuit generation | 272 s | 63 s |
| Transpilation | 47 s | 7 s |
| Execution | 69 s | 40 s |
| Transpiled depth | 689,027 | 638,237 |
| Transpiled gates | 689,127 | 604,520 |

The new transpiler produces slightly smaller circuits (7% fewer gates, 7% less
depth) while running 6x faster. This suggests improved optimization passes in
the Qiskit 2.x transpiler pipeline.

### Results: Solution Quality

The Qiskit version change **affects Newton convergence behavior**. Representative
trial-0 comparison at CFL = 1e10:

| Metric | Nov 2025 (trial 0) | Apr 2026 (trial 0) |
|---|---|---|
| Final fidelity (iter 14) | 0.959 | 0.992 |
| Final L2 error normalized | 0.203 | 0.087 |
| Final L2 error absolute | 3.0e-7 | 1.5e-2 |
| Final residual | 5.0e-6 | 0.171 |
| Residual reduction (15 iters) | ~6 orders of magnitude | ~1 order of magnitude |

Nov 2025 drives the linear system residual from 2.45 down to 5e-6 over 15
iterations — the dq correction shrinks to near-zero, indicating Newton convergence
toward the steady-state solution. Apr 2026 reduces the residual from 3.82 to only
0.17, with the dq correction plateauing around 0.015.

Despite the weaker residual convergence, the Apr 2026 run maintains higher HHL
fidelity throughout (0.992 vs 0.959 at iter 14) and a lower normalized L2 error
(0.087 vs 0.203). This apparent contradiction — better per-solve accuracy but
worse overall convergence — suggests that the transpiler changes in Qiskit 2.x
produce a different circuit realization that, while more faithful to the ideal
HHL solution, interacts differently with the nonlinear Newton iteration at
CFL = 1e10.

The practical impact depends on which
metric matters for the application: if the goal is to reach a tight residual
tolerance, the Nov 2025 stack converges faster. If the goal is a good
approximate solution in 15 iterations, both stacks produce usable results with
the Apr 2026 stack having lower normalized error.

### Code Differences: `quantum_linear_solvers` Port

The HHL circuit is generated by the `quantum_linear_solvers` library, which was
forked for the Qiskit 2.x port. The two versions:

- **Nov 2025**: `jw676/quantum_linear_solvers` (original, Qiskit 1.x)
- **Apr 2026**: `Q8020-CFD/quantum_linear_solvers` (ported fork, Qiskit 2.x)

The `hhl.py` diff reveals three categories of changes:

**1. QPE construction: `PhaseEstimation` class (both versions)**

Both forks call `PhaseEstimation(nl, matrix_circuit)` to build the QPE
sub-circuit. The `NumPyMatrix` class (unchanged between forks) builds the
controlled unitary via `scipy.linalg.expm` → `qc.unitary()`. The `UnitaryGate`
is then decomposed by the transpiler.

Both runs use `backend_type='ideal'` (AerSimulator), which hits the default
`transpile(circ, backend)` path in `quantum_tools.py` — no explicit
`optimization_level` or `seed_transpiler`. Qiskit defaults to
`optimization_level=1` in both versions. The 7% reduction in transpiled
gate count and depth (604k vs 689k gates) is therefore attributable to
Qiskit 2.3's improved unitary synthesis at the same optimization level.

**2. State preparation: `initialize` (both versions)**

Both forks use `QuantumCircuit.initialize()` for state preparation of the
right-hand-side vector. No change here.

**3. `qiskit-algorithms` removal**

The original fork imports `AlgorithmResult` from `qiskit-algorithms`. The
Q8020-CFD fork replaces this with a minimal local stub class in
`linear_solver.py`. This is a structural change only — `AlgorithmResult` is
a container class with no algorithmic behavior. It does not affect circuit
generation or execution.

**4. No changes to `_get_delta`, `ExactReciprocal`, or eigenvalue logic**

The delta calibration function, evolution time computation, and eigenvalue
inversion via `ExactReciprocal` are byte-identical between the two forks.
The mathematical core of HHL is unchanged.

**Summary**: The `quantum_linear_solvers` port contains only API-level
changes (removing deprecated Qiskit 1.x imports, stubbing out
`qiskit-algorithms`). No algorithmic or circuit-construction logic was
modified. The observed differences in transpiled circuit size and convergence
behavior originate entirely from Qiskit's internal transpiler and Aer
simulator changes between versions 1.2.4 and 2.3.1.

### Analysis: Convergence Divergence

The unchanged HHL algorithm produces functionally identical pre-transpilation
circuits in both runs. The convergence difference therefore traces to how
Qiskit decomposes and optimizes those circuits:

- **Unitary synthesis**: Qiskit 2.3 uses a different default synthesis
  strategy for `UnitaryGate`, producing a circuit with 12% fewer CX gates.
  Fewer CX gates generally improve fidelity on noisy hardware but might shift
  the statevector distribution under shot-based sampling.

- **Aer statevector sampling**: The `qiskit-aer` upgrade (0.15.1 → 0.17.2)
  includes changes to the Aer simulator's internal random number generation
  and shot sampling paths.

- **Accumulation over iterations**: At CFL = 1e10, the Newton update is
  large and the system is sensitive to the direction (not just magnitude) of
  the HHL correction vector. Small per-solve differences compound over 15
  iterations, leading to divergent convergence trajectories despite
  identical solver parameters.

The Nov 2025 stack drives the residual to 5e-6 (near machine precision for
the HHL solve), while the Apr 2026 stack plateaus at 0.17. However, the
Apr 2026 stack maintains consistently higher per-solve fidelity (~0.995 vs
~0.97), suggesting the transpiled circuit is actually more accurate per
iteration — the correction vector is simply applied with a different
effective scaling, preventing the same convergence trajectory.

### Conclusion

Upgrading from Qiskit 1.2.4 to 2.3.1 reduces per-case quantum processing
time by 3.4x on the FVM Euler 1D nozzle at 5 elements / 150k shots. The
improvement comes primarily from circuit generation (4.2x) and transpilation
(6.2x), with a more modest gain in Aer simulation execution (1.6x).

The `quantum_linear_solvers` port to Qiskit 2.x contains only API-level
changes; the HHL algorithm and circuit construction logic are identical. The
observed convergence difference at CFL = 1e10 originates from Qiskit's
internal transpiler producing a different (smaller) circuit decomposition
and from changes in Aer's shot sampling. Both stacks produce physically
reasonable nozzle flow solutions, but the Newton convergence trajectories
diverge due to accumulated per-iteration differences in the sampled HHL
correction vectors.

### Next Steps

1. **More iterations at CFL=1e10** — Run Qiskit 2.3 with 50 iterations to
   determine whether the residual is slowly converging or truly stalled.
2. **Increase shots** — Run Qiskit 2.3 at 500k shots (CFL=1e10, 15 iters)
   to test whether shot noise is the limiting factor for convergence.
3. **Lower CFL** — Run Qiskit 2.3 at CFL=1000 to reduce Newton step size,
   making each iteration less sensitive to correction vector direction errors.
